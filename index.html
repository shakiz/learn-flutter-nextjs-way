Step 01 · Foundations

Core concepts &
mental model reset

You already think in components, state, and routing. Flutter maps to all of it — but the rendering engine is completely different.

🚨
The big mindset shift: Flutter has no DOM, no CSS, no browser. It uses the Skia/Impeller graphics engine — think of it like a game engine. Every widget paints its own pixels directly onto a canvas. This is why Flutter apps look identical on every platform.

The parallel universes

🌐

Next.js world

You write JSX that maps to HTML elements. The browser's layout engine (Blink/WebKit) handles rendering. CSS positions elements, the DOM is mutated, and React reconciles virtual DOM diffs.

📱

Flutter world

You write widget trees that map to drawn shapes. Flutter's engine renders directly to a canvas — no browser, no HTML. Layout is done by Flutter's own Yoga-like constraint system.

// Build behavior, env, rewrites const nextConfig = { experimental: { appDir: true, }, images: { domains: ['api.example.com'], }, env: { API_URL: 'https://api.example.com', }, }; module.exports = nextConfig;

Step 02 · UI Building Blocks

Widgets are just
components

A widget is Flutter's version of a React component. Same idea: composable, reusable UI primitives arranged in a tree.

React (function component)
interface Props { name: string; age: number; } const UserCard = ({ name, age }: Props) => { return ( <div className="card"> <h2>{name}</h2> <p>Age: {age}</p> </div> ); };
Flutter (StatelessWidget)
class UserCard extends StatelessWidget { final String name; final int age; const UserCard({ super.key, required this.name, required this.age, }); @override Widget build(BuildContext context) { return Column(children: [ Text(name, style: TextStyle(fontSize: 20)), Text('Age: $age'), ]); } }

StatelessWidget vs StatefulWidget

🔒

StatelessWidget

Equivalent to a pure function component — receives props (constructor params), returns UI. No internal state. Always re-renders the same output for the same props.

like React.memo use const
🔄

StatefulWidget

Has a separate State class that holds mutable state. Calling setState() schedules a rebuild — exactly like calling a useState setter in React.

like useState class
React useState
const Counter = () => { const [count, setCount] = useState(0); return ( <div> <p>Count: {count}</p> <button onClick={() => setCount(c => c + 1)} > Increment </button> </div> ); };
Flutter StatefulWidget
class Counter extends StatefulWidget { const Counter({super.key}); @override State<Counter> createState() => _CounterState(); } class _CounterState extends State<Counter> { int count = 0; @override Widget build(BuildContext ctx) { return Column(children: [ Text('Count: $count'), ElevatedButton( onPressed: () => setState(() => count++), child: Text('Increment'), ), ]); } }

Step 03 · Platform

Lifecycle on
Android & iOS

Flutter sits on top of native Android and iOS lifecycles. Understanding both layers is key to handling background state, notifications, and memory correctly.

⚡
Two lifecycle layers: Flutter has its own widget lifecycle (like React's component lifecycle), AND it exposes the native Android/iOS app lifecycle. You usually only need Flutter's layer — but background tasks and foreground service detection require the native one.

Flutter widget lifecycle

React lifecycle (class analog)
// Mounting componentDidMount() // after render // Updating componentDidUpdate() // after re-render shouldComponentUpdate() // Unmounting componentWillUnmount() // cleanup // Hooks equivalent useEffect(() => { // mount logic return () => { // cleanup }; }, []);
Flutter State lifecycle
// Called once at creation initState() // Called when widget config changes didUpdateWidget(oldWidget) // Called every build (like render) build(context) // Called when dependencies change didChangeDependencies() // Cleanup — like useEffect cleanup dispose() // Called on hot reload reassemble()

Native app lifecycle

🤖 Android lifecycle

01
onCreate()
App process created. Flutter engine initializes. Like the very first page load.
02
onStart()
Activity becomes visible but not interactive yet.
03
onResume()
App is in the foreground, fully interactive. → Flutter: resumed
04
onPause()
Partially obscured (notification shade, split screen). → Flutter: inactive
05
onStop()
App moved to background (user switches apps). → Flutter: paused
06
onDestroy()
Activity destroyed. System kills the process. → Flutter: detached

🍎 iOS lifecycle

01
Not Running
App is not launched. Process doesn't exist.
02
Inactive
App in foreground but receiving no events (phone call, gesture). → Flutter: inactive
03
Active
Running and receiving events. Fully interactive. → Flutter: resumed
04
Background
App in background, still running code. Briefly execute tasks before suspension. → Flutter: paused
05
Suspended
In memory but no code running. iOS can terminate at any time without notice. → Flutter: detached

Listening to app lifecycle in Flutter

Web equivalent (visibility API)
useEffect(() => { const handler = () => { if (document.hidden) { pauseVideo(); } else { resumeVideo(); } }; document.addEventListener( 'visibilitychange', handler ); return () => document.removeEventListener( 'visibilitychange', handler ); }, []);
Flutter WidgetsBindingObserver
class _MyState extends State<MyWidget> with WidgetsBindingObserver { @override void initState() { super.initState(); WidgetsBinding.instance .addObserver(this); } @override void didChangeAppLifecycleState( AppLifecycleState state ) { if (state == AppLifecycleState.paused) { pauseVideo(); } else if (state == AppLifecycleState.resumed) { resumeVideo(); } } @override void dispose() { WidgetsBinding.instance .removeObserver(this); super.dispose(); } }
Step 04 · Platform

Routing &
navigation

Next.js file-based routing is more magical. Flutter's routing is more explicit — closer to React Router v6 than the Next.js App Router.

Next.js App Router
// File system = routes automatically app/ page.tsx → / about/page.tsx → /about users/ [id]/page.tsx → /users/:id // Navigate programmatically import { useRouter } from 'next/navigation'; const router = useRouter(); router.push('/about'); router.push(`/users/${id}`);
go_router (recommended)
// Explicit route declaration final router = GoRouter( routes: [ GoRoute( path: '/', builder: (ctx, state) => HomePage(), ), GoRoute( path: '/users/:id', builder: (ctx, state) { final id = state.pathParameters['id']; return UserPage(id: id!); }, ), ], ); // Navigate context.go('/about'); context.push('/users/$id');
🔁
go() vs push(): context.go() replaces the stack (like router.replace in Next.js). context.push() adds to the back-navigation stack. Mobile users expect a back button — use push() for drilldown screens.
Passing params & query
// URL params const { id } = useParams(); // Query string const sp = useSearchParams(); const tab = sp.get('tab'); // Navigate with query router.push('/users?tab=orders');
go_router params
// Path params from GoRouterState final id = state.pathParameters['id']; // Query params final tab = state.uri.queryParameters['tab']; // Navigate with query context.push('/users?tab=orders'); // Pass objects via extra context.push('/detail', extra: userObject);
Step 05 · Data & State

State
management

Flutter has a rich ecosystem of state management solutions. If you know React Context + Zustand, you'll feel at home with Provider + Riverpod.

Local state

setState

Built-in to StatefulWidget. Best for UI-only state: form inputs, toggles, tab selection.

≈ useState
Shared / global

Riverpod

The modern recommended solution. Type-safe providers, auto-dispose, testable. Works outside BuildContext.

≈ Zustand / Jotai
Reactive streams

BLoC / Cubit

Event-driven, stream-based. Great for complex business logic and large teams. More boilerplate but very predictable.

≈ Redux + RTK

Riverpod in practice

Zustand store
import { create } from 'zustand'; const useUserStore = create((set) => ({ user: null, isLoading: false, fetchUser: async (id) => { set({ isLoading: true }); const data = await getUser(id); set({ user: data, isLoading: false }); }, })); // In component const { user, fetchUser } = useUserStore();
Riverpod provider
// Define provider final userProvider = AsyncNotifierProvider< UserNotifier, User >(UserNotifier.new); class UserNotifier extends AsyncNotifier<User> { @override Future<User> build() => fetchUser(); Future<void> refresh() async { state = const AsyncLoading(); state = await AsyncValue.guard(fetchUser); } } // In widget — like useSelector final userAsync = ref.watch(userProvider); userAsync.when( data: (u) => Text(u.name), loading: () => CircularProgressIndicator(), error: (e, _) => Text('Error: $e'), );
🎯
The Riverpod mental model: Think of ref.watch() as useSelector() — it subscribes and rebuilds on change. ref.read() is a one-time read without subscribing (like using the store outside a component). ref.listen() is like a useEffect that fires on state change.
Step 06 · Data & State

Data
persistence

Mobile apps need offline-first storage. Flutter has a rich ecosystem: from simple key-value to full SQL, all with pub.dev packages.

Use case Flutter package Next.js / web equivalent Notes
Simple key-value shared_preferences localStorage Sync-like API. Stores strings, bools, ints. Great for settings, tokens.
Secure storage flutter_secure_storage httpOnly cookie Uses Android Keystore / iOS Keychain. For JWT tokens, passwords.
Local SQL database sqflite SQLite / Postgres (server-side) Full SQLite on device. Raw SQL queries. Use drift for ORM on top.
ORM / type-safe DB drift Prisma / Drizzle Code-gen schema, typed queries, reactive streams. The Prisma of Flutter.
NoSQL / document isar Firestore / MongoDB Very fast. Schema defined in Dart. Great for offline-first apps.
File storage path_provider fs (Node.js) Gets app-specific directories (documents, temp, cache). Read/write files.
Caching / memoization hive SWR / React Query cache Fast key-value box. No codegen needed. Boxes = tables.

Step 07 · Tooling

pubspec.yaml
deep dive

The single config file that replaces package.json, next.config.js, .nvmrc, and your asset pipeline — annotated for Next.js devs.

Three Next.js files
// package.json { "name": "my-app", "version": "1.0.0", "dependencies": { ... }, "scripts": { ... } } // .nvmrc 20.11.0 // next.config.js module.exports = { images: { domains: [...] }, env: { API_URL: '...' }, };
One pubspec.yaml
# ← package.json: name, version name: my_app description: My Flutter app version: 1.0.0+1 # ← .nvmrc: runtime version constraint environment: sdk: ">=3.3.0 <4.0.0" # ← package.json dependencies dependencies: flutter: { sdk: flutter } riverpod: ^2.5.1 go_router: ^13.2.0 # ← package.json devDependencies dev_dependencies: flutter_test: { sdk: flutter } build_runner: ^2.4.8 # ← next.config.js: framework config flutter: uses-material-design: true assets: # ← /public/* - assets/images/ - assets/json/ fonts: # ← @font-face - family: Syne fonts: - asset: assets/fonts/Syne.ttf weight: 700
🔢
The version number trick: 1.0.0+1 — the part before + is the user-facing version (semver), the part after is the build number. App stores (Play Store, App Store) use the build number for internal versioning. You increment it on every release. No equivalent in web dev — unique to mobile distribution.

Asset handling difference

🌐

Next.js — automatic

Anything in /public is automatically served. Import images with next/image. No declaration needed.

📦

Flutter — explicit

Every asset folder must be declared in pubspec.yaml. Undeclared assets throw a runtime error. Use trailing slash for whole directories.

Step 08 · Tooling

Dart for
JS developers

Dart feels like TypeScript with stricter types and no build toolchain anxiety. Most JS patterns translate directly — with some pleasant surprises.

TypeScript
// Types const name: string = "Sakhawat"; let count: number = 0; // Optional chaining const city = user?.address?.city; // Null coalescing const label = value ?? 'default'; // Async/await const fetchUser = async (id: string) => { const res = await fetch(`/api/user/${id}`); return res.json(); }; // Destructuring const { name, age } = user; // Spread const merged = { ...defaults, ...overrides };
Dart
// Types (var = inferred like const/let) var name = "Sakhawat"; // inferred String int count = 0; // Optional chaining — same! final city = user?.address?.city; // Null coalescing — same! final label = value ?? 'default'; // Async/await — same syntax! Future<User> fetchUser(String id) async { final res = await dio.get('/user/$id'); return User.fromJson(res.data); } // No destructuring — use named vars final name = user.name; final age = user.age; // Spread in collections final merged = { ...defaults, ...overrides };

The Dart features JS doesn't have

🏷️

Named parameters

Flutter widgets use named params by default. required makes them mandatory. This is why Flutter code reads so naturally:

Text(
  'Hello',
  style: TextStyle(fontSize: 18),
  textAlign: TextAlign.center,
)
🔐

Sound null safety

Dart's null safety is compile-time enforced — stricter than TypeScript's. String? can be null. String cannot. The compiler will not let you forget.

String? maybeNull = null; // OK
String mustExist = 'hi'; // OK
mustExist = null; // ❌ compile error
🔀

Pattern matching (Dart 3)

Dart 3 added switch expressions and pattern matching — like TypeScript discriminated unions but built into the language syntax:

final msg = switch (status) {
  Status.ok => 'All good',
  Status.error => 'Failed',
  _ => 'Unknown',
};
📋

Extension methods

Add methods to any type — even SDK types — without subclassing. Like prototype extensions but type-safe and scoped to the import:

extension StringX on String {
  bool get isEmail => contains('@');
}

// Usage
'user@email.com'.isEmail; // true
🎓
Closing thought: Next.js abstracts the browser. Flutter abstracts the OS. Same philosophy, different target. Your instinct for component composition, state isolation, and async data fetching will carry over almost 1:1. The learning curve is mostly Dart syntax and mobile platform quirks — not a new way of thinking.
