Step 01 · Foundations

Core concepts &
mental model

You already think in components, state, and routing. Flutter maps to all of it — but the rendering engine is completely different, and that changes everything.

🚨
The big shift: Flutter has no DOM, no CSS, no browser. It uses the Skia / Impeller graphics engine — like a game engine. Every widget paints its own pixels directly onto a canvas. This is why Flutter apps look identical on every platform.

The parallel universes

🌐

Next.js world

You write JSX that maps to HTML elements. The browser's layout engine handles rendering. CSS positions elements, the DOM is mutated, React reconciles virtual DOM diffs.

📱

Flutter world

You write widget trees that map to drawn shapes. Flutter's engine renders to a canvas — no browser, no HTML. Layout is handled by Flutter's own constraint system.

Step 03 · Platform

App lifecycle
in depth

Two lifecycle layers exist simultaneously: the native OS layer (Android Activity / iOS UIApplicationDelegate) and Flutter's own widget layer. Both matter — for different problems.

⚡
Two layers, one app. Flutter surfaces the native lifecycle through AppLifecycleState enum. But the OS also has finer-grained states (partial obscurity, system dialog overlay) that Flutter collapses. Knowing both helps you understand why certain callbacks fire unexpectedly.

Flutter widget lifecycle — the React comparison

React hooks lifecycle
useEffect(() => { // Mount — equivalent to initState fetchData(); const sub = subscribe(); return () => { // Unmount — equivalent to dispose sub.unsubscribe(); }; }, []); useEffect(() => { // Dep changed — didUpdateWidget refetch(userId); }, [userId]);
Flutter State lifecycle
void initState() { super.initState(); // Runs once on mount. Sync only. fetchData(); _sub = stream.listen(_onEvent); } void didUpdateWidget(MyWidget old) { super.didUpdateWidget(old); // Parent rebuilt with new props if (old.userId != widget.userId) refetch(widget.userId); } void dispose() { // Cleanup — cancel streams, timers _sub.cancel(); _ctrl.dispose(); super.dispose(); }

Native app states — Android

🟢  FOREGROUND PHASE — user can see & interact with the app
1
onCreate()
App process is born. Flutter engine boots, Dart VM starts, main() runs. Equivalent to your server-side getServerSideProps executing on cold start. Expensive work here delays your splash screen.
Flutter: detached → inactive
2
onStart()
Activity is visible but not interactive yet — imagine a page that's painted but JS hasn't hydrated. Brief. Usually milliseconds.
3
onResume()
App is in the foreground, fully interactive, receiving touch events. This is your "app is live" state. Resume animations, restart timers, reconnect WebSockets here.
Flutter: AppLifecycleState.resumed
⚪  TRANSITION / PARTIAL VISIBILITY — system UI overlays the app
4
onPause()
App is partially obscured — notification shade pulled down, split-screen resize, incoming call screen overlaid, or screen about to turn off. The app is still visible but is losing focus. This is where you pause video playback, stop camera preview, save lightweight state. Must return quickly — Android can kill the process if onPause() blocks.
Flutter: AppLifecycleState.inactive (also: hidden on Android 14+)
⬛  BACKGROUND PHASE — user switched away, app is invisible
5
onStop()
App is fully invisible — user pressed Home or switched apps. The process still lives in memory. You have a short grace period to release heavy resources: stop music streams, cancel background location if not needed, flush pending analytics. Think of this as the equivalent of a tab going visibilityState = "hidden".
Flutter: AppLifecycleState.paused
6
onRestart()
User comes back to the app from the stopped state (not killed). onStart() → onResume() follow. Restore any state you saved in onStop().
🔴  TERMINATION — process dies
7
onDestroy()
Activity is destroyed — either the user explicitly closed the app, or Android killed it for memory. You cannot rely on this running — Android can kill the process between onStop() and onDestroy() without calling it. Always save critical state in onPause() or onStop(), not here.
Flutter: AppLifecycleState.detached

Native app states — iOS

🍎
iOS is stricter than Android. iOS can terminate a suspended app with zero warning — no applicationWillTerminate is called. You have ~5 seconds when moving to background before suspension. Background execution requires explicit capability declarations (Background Fetch, Push Notifications, etc.) otherwise your Dart code stops running the moment the user presses Home.
🟢  ACTIVE STATES
1
Active
App is in foreground, fully interactive. Equivalent to Android's onResume. Normal operating state.
Flutter: resumed
2
Inactive
Transitional state — app is running but not receiving events. Happens during: incoming phone call overlay, app switcher gesture (user swiping up), Control Center / Notification Center pulled down, Siri activation. Pause game loops and real-time updates here — the user isn't interacting but the app is still visible.
Flutter: inactive
⬛  BACKGROUND / SUSPENDED
3
Background (executing)
App has moved to background. You get a short execution window (~5 seconds for most tasks, longer for declared background modes like audio, location, voip). Use beginBackgroundTask in native code to extend this. Flutter's Dart isolate may be paused — use native platform channels for guaranteed background execution.
Flutter: paused
4
Suspended
App is in memory but no code runs. iOS decides when to terminate. Memory pressure → your app is killed silently. On next launch, Flutter calls initState() fresh — there is no "resume from suspended" event. This is why persistent state (SharedPreferences, SQLite) is the only reliable storage across app launches.
iOS can terminate at any time from here

Listening to lifecycle in Flutter

Web Page Visibility API
useEffect(() => { const onVisibility = () => { if (document.hidden) { pauseMedia(); analytics.flush(); } else { resumeMedia(); socket.reconnect(); } }; document.addEventListener( 'visibilitychange', onVisibility ); return () => document.removeEventListener( 'visibilitychange', onVisibility ); }, []);
WidgetsBindingObserver mixin
class _MyState extends State<MyPage> with WidgetsBindingObserver { @override void initState() { super.initState(); WidgetsBinding.instance.addObserver(this); } @override void didChangeAppLifecycleState( AppLifecycleState state ) { switch (state) { case AppLifecycleState.resumed: resumeMedia(); socket.reconnect(); case AppLifecycleState.inactive: pauseGameLoop(); case AppLifecycleState.paused: pauseMedia(); analytics.flush(); case AppLifecycleState.detached: // engine detaching — unreliable } } @override void dispose() { WidgetsBinding.instance.removeObserver(this); super.dispose(); } }
✅
Practical lifecycle cheat sheet: Save lightweight UI state in paused (user went background). Save critical data in inactive (might be killed soon). Restore connections in resumed. Never rely on detached — it may not fire on iOS at all.
Step 04 · Platform

Navigation
in depth

Flutter's navigation is a physical back-stack of screens. No file-system magic — but it gives you fine-grained control over back gestures, deep links, nested navigators, and modal flows that Next.js can't express natively.

📚
The core mental model: Flutter maintains a stack of routes. Push adds a screen on top. Pop removes the top screen and returns to the one below. The OS back button / back gesture always pops. go_router wraps this with URL-style path matching — the stack is reconstructed from the URL, just like a browser.

go() vs push() — the critical difference

↗️

context.push()

Adds a new route on top of the stack. Back button returns to the previous screen. Use for drilldown flows: list → detail → edit. The user expects to be able to go back.

Stack growsBack button works
↪️

context.go()

Replaces the entire stack with the new route tree. Back button has nothing to go back to. Use for authentication flows (after login → go Home), tab switches, and error recovery where back would be confusing.

Stack replacedNo back

Nested navigators — tabs with their own back-stacks

🗂️
This is the biggest difference from web routing. Mobile apps typically have a bottom tab bar where each tab maintains its own independent stack. Navigate deep inside the "Shop" tab, switch to "Profile" tab, come back — you're still deep in "Shop". This is nested navigation and it has no direct equivalent in Next.js App Router.

Returning data from a pushed route

💡
A pattern Next.js can't do natively: In Flutter you can push a screen and await the result when the user pops back. Like a modal that returns a value. No URL params needed.
Web workaround (URL state)
// No native "return value" from route // Must use URL params or global state router.push('/pick-color?from=settings'); // ColorPicker reads ?from and // saves to localStorage / zustand // then router.back() // Settings page polls store on mount useEffect(() => { const picked = colorStore.getPicked(); if (picked) setColor(picked); }, []);
Flutter — await a route result
// Push and await the popped value final color = await Navigator.push<Color>( context, MaterialPageRoute( builder: (_) => ColorPickerScreen(), ), ); // color is non-null if user picked if (color != null) setState(() => _color = color); // ColorPickerScreen pops with value ElevatedButton( onPressed: () => Navigator.pop(context, Colors.blue), child: Text('Pick blue'), )

Step 07 · Tooling

pubspec.yaml
deep dive

One file replaces package.json, next.config.js, .nvmrc, and your asset pipeline — annotated for Next.js developers.

Three separate Next.js files
// package.json { "name": "my-app", "version": "1.0.0", "dependencies": { "next": "^14" } } // .nvmrc 20.11.0 // next.config.js module.exports = { images: { domains: ['cdn.x.com'] }, env: { API_URL: '...' }, };
pubspec.yaml — annotated
# ← package.json name + version name: my_app version: 1.0.0+1 # semver + build# # ← .nvmrc (Dart SDK version) environment: sdk: ">=3.3.0 <4.0.0" # ← dependencies dependencies: flutter: { sdk: flutter } go_router: ^13.2.0 dio: ^5.4.0 # ← devDependencies dev_dependencies: flutter_test: { sdk: flutter } build_runner: ^2.4.8 # ← next.config.js (framework config) flutter: uses-material-design: true assets: # ← /public - assets/images/ - assets/json/ fonts: # ← @font-face - family: Fraunces fonts: - asset: assets/fonts/Fraunces.ttf weight: 700
🔢
The version number trick: 1.0.0+1 — before + is the user-facing semver, after is the build number. App stores use the build number for internal versioning. Increment it on every release. No web equivalent — unique to mobile distribution.
Step 08 · Tooling

Dart for
JS developers

Dart feels like TypeScript with stricter null safety and no build toolchain anxiety. Most JS patterns translate directly — with some pleasant surprises.

TypeScript
const name: string = "Sakhawat"; const city = user?.address?.city; const label = value ?? 'default'; const fetch = async (id: string) => { const res = await getUser(id); return res.json(); }; // Destructuring const { name, age } = user; // Spread const merged = { ...a, ...b };
Dart
final String name = 'Sakhawat'; final city = user?.address?.city; // same! final label = value ?? 'default'; // same! Future<User> fetch(String id) async { final res = await dio.get('/users/$id'); return User.fromJson(res.data); } // No destructuring — named vars final name = user.name; // Spread works in collections final merged = { ...a, ...b };

The Dart features JS doesn't have

🏷️

Named parameters

Flutter widgets use named params by default. required makes them mandatory. This is why Flutter widget calls read like English:

Text('Hello', style: TextStyle(fontSize: 18), textAlign: TextAlign.center)
🔐

Sound null safety

Stricter than TypeScript. String? can be null. String cannot — the compiler enforces it, no runtime surprises.

String? maybe = null; // OK
String must = null; // ❌ compile error
🔀

Pattern matching (Dart 3)

Switch expressions with exhaustive matching — like TypeScript discriminated unions but built into the language:

final msg = switch (status) {
  Status.ok => 'All good',
  Status.error => 'Failed',
  _ => 'Unknown',
};
📋

Extension methods

Add methods to any type — even SDK types — without subclassing. Scoped to import.

extension StringX on String {
  bool get isEmail => contains('@');
}
'user@email.com'.isEmail; // true
🎓
Closing thought: Next.js abstracts the browser. Flutter abstracts the OS. Same philosophy, different target. Your instinct for component composition, state isolation, and async data fetching carries over almost 1:1. The learning curve is Dart syntax + mobile platform quirks — not a new way of thinking.
